# Implement: Upgrade & Adapt

## The Implementation

The dependency pin in `backend/functions/package.json` moved from `^13.0.0` to the exact `17.2.0`, and `pnpm install` resolved it without peer-dependency conflicts. The compiler greeted the bump with silence — zero errors — because the existing `as any` cast in `tv-oauth.ts` had been suppressing the only type-level issue, and the rest of the codebase (`fetch.ts`, `innertube.ts`, `innertube-sync.ts`) uses the youtubei.js API surfaces that the upstream changelog confirmed are stable across the four majors.

The substantive work was all in `tv-oauth.ts`. The v17 `ClientType` enum now encodes the correct YouTube wire names directly (`ClientType.TV = "TVHTML5"`), so the `as any` cast was replaced with `client_type: ClientType.TV` and the ten-line session-context patch — the hand-rolled fix for v13's incorrect wire-name emission — was deleted entirely. The local `ICache` interface stayed as a structural copy: v17 exports it only through a type-namespace alias (`Types.ICache`), not as a top-level named export, making a direct import non-idiomatic. The interface body is bit-for-bit identical to v17's definition, so structural typing keeps the `FirestoreOAuthCache` class in sync without coupling to the library's internal export topology.

All four ops scripts (`scripts/*.mjs`) passed `node --check` unchanged. The plan's Assumption 3 was confirmed: the scripts' context-patch lines remain in place as harmless guards that are now technically no-ops in v17.

The full vitest suite — 17 test files, 104 assertions — ran green against the Firestore emulator in 45 seconds. The transcript blob/pointer contract is preserved bit-for-bit. The 400s in production are not fixed by this slice; that is expected and by design — this slice's job was to establish a current, compiling, fully-tested baseline.

## Notes on Design Choices

- **`ICache` kept as local copy, not imported.** v17 exports `ICache` only via `export type * as Types from '../types/index.js'` (i.e., `Types.ICache`), not as a direct top-level type export. The structural duplicate is identical to v17's definition, so TypeScript's structural typing ensures the `FirestoreOAuthCache` implementation stays compatible. If a future v18+ re-exports `ICache` at the top level, removing the local copy becomes a one-liner.

- **Context patch removed (conditional per plan, evidence confirmed).** The plan instructed to remove the patch only if v17 evidence confirmed it sends the correct wire names natively. Evidence from `Session.js` line 18 (`TV: "TVHTML5"`) and the CLIENTS lookup (line 97: `NAME === client_name`) confirms that `ClientType.TV` drives `clientName = "TVHTML5"` in v17 session initialization — the patch is definitively a no-op on v17.

- **Ops scripts unchanged.** The scripts pass `client_type` as a key-string (`"TV"`, `"TV_SIMPLY"`) which in v17 would miss the CLIENTS enum-value lookup. However, all four scripts also apply the context patch immediately after `signIn()`, overriding `clientName` to the correct wire name. Net runtime behavior is correct. The plan's Assumption 3 (preserve script context patches) was confirmed correct.

- **Build was zero-error before any changes.** The `as any` cast in the old `tv-oauth.ts` suppressed the only type-level discrepancy. Steps 5–7 were confirmed no-ops; only step 4 (tv-oauth.ts) required code changes.

## Known Risks / Caveats

- **Ops scripts use enum key-strings as `client_type`.** `test-tv-oauth.mjs` passes `"TV"`, `"TV_SIMPLY"`, `"TV_EMBEDDED"` as `client_type`, which are enum keys, not values (`"TVHTML5"`, `"TVHTML5_SIMPLY"`, `"TVHTML5_SIMPLY_EMBEDDED_PLAYER"`). In v17, the CLIENTS lookup finds entries by `NAME` (the value), so the initial session config may default to WEB client version. The context patch in each script corrects `clientName` and `clientVersion` after `signIn()`, so runtime behavior is correct — but the mismatch means the v17 SDK won't auto-select the right client version during `getSessionData`. This is harmless for ops/diagnostic scripts.
